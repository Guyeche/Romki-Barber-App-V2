# Schedule Breaks & Time Blocks — Design

**Date:** 2026-07-06
**Branch:** feature/new-ui-design
**Status:** Approved, ready for implementation plan

## Problem

The barber cannot carve breaks out of the middle of his working day. The schedule
models each day as a single unbroken block (`work_schedule.start_time` →
`work_schedule.end_time`), and slots are generated by walking that block in 20-minute
steps. The only existing "off" mechanism, `blocked_days`, blocks **whole days** only.
There is no way to punch a hole (lunch, an errand, "stepping out for an hour") inside
an otherwise-open day.

The barber needs three things:
1. A **recurring** daily break (e.g. lunch 13:00–14:00 every working day).
2. **One-off** blocks on a specific date+time (e.g. "this Thursday I'm out 15:00–16:00").
3. A fast **"block from now"** action for when something comes up mid-shift.

## Approach (chosen)

**Unified "blocks" model (subtract holes).** Working hours stay exactly as they are
(one open block per day). A new `schedule_blocks` table represents time to *remove*.
The slot generator builds the day as it does today, then drops any slot that overlaps
a block. One model powers all three features; the change is additive rather than a
rewrite of the existing hours logic or admin UI.

Rejected alternatives:
- **Multi-segment working hours** (split each day into intervals): pure for the
  recurring break, but one-off blocks and "block from now" still need a separate
  mechanism, and it is the most invasive rewrite of schema + admin UI.
- **Extend `blocked_days` with optional times:** smallest change, but has no home for
  the recurring break and overloads one table with two meanings.

## Data model

New table `schedule_blocks`. Each row is a chunk of time to remove from an otherwise-open
day. A block is **either** recurring (tied to a weekday) **or** one-off (tied to a date),
never both.

```
schedule_blocks
├─ id           BIGINT       PK, auto-generated
├─ day_of_week  INT          NULL      -- 0–6 (0=Sunday). Set for RECURRING breaks.
├─ date         DATE         NULL      -- YYYY-MM-DD. Set for ONE-OFF blocks.
├─ start_time   TIME         NOT NULL  -- e.g. 13:00
├─ end_time     TIME         NOT NULL  -- e.g. 14:00
├─ reason       TEXT         NULL      -- optional label ("Lunch", "Dentist")
└─ created_at   TIMESTAMPTZ  default now()

CHECK: exactly one of (day_of_week, date) is non-null
CHECK: end_time > start_time
```

- Recurring vs one-off is **derived** (`date IS NOT NULL` = one-off) — no `kind` column
  to keep in sync.
- `blocked_days` (whole-day off) stays untouched.
- **Multiple blocks per day are supported and expected** — each block is its own row.
  A date may have many one-off blocks (e.g. 11:00–12:00 and 15:00–16:00). v1 limits only
  the *recurring* break to one per weekday (that is "lunch"); the table supports more
  later without change.

### Overlap semantics

A block is a half-open interval `[start_time, end_time)`. A 20-minute slot beginning at
`slotStart` is removed when it overlaps a block:

```
slotStart < block.end_time  AND  slotStart + 20min > block.start_time
```

Example: a 13:00–14:00 break removes the 13:00, 13:20, 13:40 slots and leaves 14:00
bookable. Overlapping blocks are harmless — they just remove the same slots twice.

## Slot generation & booking validation

Blocks are subtracted in **two** places. The first is UX; the second is the real guard.

### 1. Slot generation (cosmetic) — `components/DateTimePicker.tsx`
After the raw 20-minute slots are built for a date (currently ~lines 139–191):
- Fetch blocks for that date: recurring rows where `day_of_week = date.getDay()` **plus**
  one-off rows where `date = selectedDate`.
- Drop any slot overlapping a block (interval test above).
- Runs alongside the existing "remove booked times" and "remove past times" filters — one
  more filter pass, same pattern.
- Blocks are read through the schedule API (extend `/api/schedule` GET, or a small
  `/api/blocks?date=`) so the client stays a thin reader.

### 2. Booking validation (the real lock) — `app/actions.ts`, `bookAppointment()`
Where it already checks `blocked_days` (~lines 140–153):
- Before the INSERT, query `schedule_blocks` for the requested weekday **and** date, and
  reject if the requested `time` falls inside any block.
- New error key `booking.timeUnavailable` ("That time is no longer available").
- **Why essential:** the client filter is cosmetic. A stale booking page, or a break
  created *after* a customer loaded the page, must still be rejected at write time. The
  DB `UNIQUE(date,time)` does not help because a break is not a booking — this server
  check is what enforces it.

### Non-goal: existing appointments are never touched
Creating a break does **not** cancel or move existing appointments. If someone booked
13:20 and the barber later adds a 13:00–14:00 lunch, that appointment stays and he still
sees it; the break only stops *new* bookings in that window. Cancelling is a separate,
manual action.

## Admin UI touchpoints

### A. Recurring break — existing Schedule page (`admin/schedule/ScheduleForm.tsx`)
- Each active weekday row gains an optional **break Start / End** pair (same 20-minute
  dropdowns as Open/Close), shown only when the day is active. Empty = no break.
- An **"Apply lunch to all active days"** helper sets one range across all active days.
- The page's existing **Save** button writes hours *and* recurring breaks together.
  Backend: extend `/api/schedule` POST to upsert the `day_of_week` rows in
  `schedule_blocks` (delete + reinsert the recurring set, mirroring the hours upsert).
- v1: one break per day.

### B. One-off block — new "Time Off" page (`admin/time-off`)
- Form: **Date · Start · End · Reason (optional)** → Add.
- Below: a list of upcoming one-off blocks, each with **Delete**.
- Built so a whole-day toggle (managing `blocked_days`) can slot in later; v1 scoped to
  timed blocks.
- Backend: `/api/blocks` (POST create, GET list, DELETE remove) operating on one-off
  (`date`-set) rows.

### C. Quick "block from now" — button on the admin dashboard
- A prominent **"Block time now"** button opens a small sheet with presets:
  **20 min · 1 hour · Rest of today** + optional reason.
- Creates a one-off block for **today** starting at the next 20-minute boundary
  (local/Israel time, matching how the picker filters past slots).
- "Rest of today" ends at that day's `end_time` from `work_schedule`.
- Can be tapped multiple times a day → multiple independent blocks.

All three touchpoints write to the same `schedule_blocks` table via the same API, so they
behave identically for customers.

## Edge cases

- **Already-booked slots inside a new break:** kept (see non-goal). The Time Off page and
  the quick-block sheet show a note when the chosen window overlaps existing appointments,
  e.g. *"⚠ 1 booking already exists in this window — it will be kept,"* so the barber can
  decide to honor or cancel it separately.
- **"Rest of today" end:** the day's `end_time` from `work_schedule`.
- **Past / near-now:** quick-block starts at the next 20-minute boundary using local time.
- **Closed / out-of-window days:** blocks on them simply have no effect — no special
  handling.

## i18n

New keys in both `messages/en.json` and `messages/he.json`:
- `admin.timeOff.*` — Time Off page (title, date, start, end, reason, add, delete, empty
  state, overlap warning).
- `admin.blockNow.*` — dashboard button + sheet (button label, presets `20min` / `1hour` /
  `restOfDay`, reason, confirm).
- `admin.schedule.break*` — recurring break labels + "apply to all" helper.
- `booking.timeUnavailable` — booking rejection error.

Hebrew RTL is already handled by the existing admin layout.

## Out of scope (v1)

- More than one recurring break per weekday.
- Whole-day `blocked_days` management UI (page is built to accommodate it later).
- Per-service durations / variable slot lengths.
- Auto-cancelling or notifying customers whose existing bookings fall in a new break.
